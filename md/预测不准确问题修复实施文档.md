## 1. 修改目标

当前问题不是随机误差，而是 `semantic-data-LLM-only` 和 `raw-data-LLM+RAG` 明显偏向预测 `S`，尤其 RAG 检索几乎只命中 `feature_card`，没有真正注入行为规则。修复目标如下：

1. 三组主实验公平可比：同一个 split、同一个 `k=5`、同一个测试集、同一个模型、同一个温度、同一个 JSON schema。
2. `semantic-risky-old` 保留为旧语义化对照；`semantic-neutral-fixed` 作为修复后的正式 semantic 实验。
3. RAG 从全局 top-k 改为分桶检索，确保每条样本都能检索到 `behavior_rule`。
4. 规则库写入 `related_label`，但 prompt 中明确说明它不是直接判决，只是规则倾向。
5. 加入排除 test 后训练池统计，让 `Cipher`、`Binder`、反射、`mount` 等“看起来危险但 Drebin 中常偏 B”的特征不再被单向渲染为恶意证据。
6. 最终报告能产出 PPT 要求的 `rules.csv`、`prompt.md`、`results.xlsx`、`figures/` 和错误案例分析材料。

## 3. P0：LLM 调用和 prompt 去歧义

### 3.1 `call_ollama()` 支持 `num_ctx=12288`

目标文件：`llm_only_fewshot.py`

把函数签名从：

```python
def call_ollama(model: str, prompt: str, temperature: float, request_timeout: float) -> str:
```

改为：

```python
def call_ollama(
    model: str,
    prompt: str,
    temperature: float,
    request_timeout: float,
    num_ctx: int = 12288,
) -> str:
```

`client.chat()` 的 `options` 改成：

```python
options={"temperature": temperature, "num_ctx": num_ctx}
```

如果走 `except TypeError` fallback，也要带同样的 `options`。所有调用方必须传入 `args.num_ctx`。

涉及 CLI：

| 文件 | 新增参数 |
|---|---|
| `llm_only_fewshot.py` | `--num-ctx`, `type=int`, `default=12288` |
| `llm_feature_expr_fewshot.py` | 同上 |
| `llm_rag_raw_fewshot.py` | 同上 |

metrics/run config 中也要记录：

```json
"num_ctx": 12288
```

显存安全策略：本机 8 GB 专用显存，`num_ctx=4096` 时实测 3.9 GB；估算 `num_ctx=12288` 总占用约 5.0 GB，留约 3 GB 余量。如果在 30 条 smoke 中观察到 Ollama OOM 或 `request_timeout` 异常飙升，将 `--num-ctx` 临时降为 `8192` 后重跑，并在 metrics 中记录实际值。不要使用低于 8192 的设置，否则 semantic-neutral-fixed 长 prompt 会被截断回到本次问题的初始状态。

目标文件：`llm_only_fewshot.py`（prompt 模板 + `infer_label_from_text()`）、`llm_rag_raw_fewshot.py`（`build_rag_prompt()` 中 prompt 模板）。

**Prompt 模板 before → after diff（统一写法）**：

Before（旧）：

After（新）：

```text
Allowed labels:
- B means benign app.
- S means malware app.

Important mapping rule:
- If your answer is benign app, output "pred_label":"B".
- If your answer is malware app, output "pred_label":"S".
```

**`infer_label_from_text()` 清理 hunk 清单**（`llm_only_fewshot.py` lines 186-243）：

需要删除的项：

1. `malware_exact` 集合（约 lines 199-208）：删除 `"suspicious"`、`"suspicious malicious"`、`"privacy invasive"` 三个字符串。保留 `"malware"`、`"malicious"`、`"trojan"`、`"adware"`、`"spyware"`、`"ransomware"` 这些**严格恶意类术语**。
2. `malware_words` 正则（约 line 227）：从交替分支中删除 `suspicious`、`privacy[-\s]?invasive` 两项。
3. 弱 fallback 块（约 lines 235-238）：删除把单独出现的 `suspicious` 直接映射成 `S` 的逻辑；保留 `pred_label`/`label` 显式字段解析路径。

1. 严格 JSON 字段 `pred_label` 或 `label` 值为 `S`/`malware`/`malicious`/`trojan` 等恶意类术语。

## 4. P0：诊断日志和 evidence 合法性

### 4.1 记录 prompt 长度和 active feature 数

在 `llm_feature_expr_fewshot.py` 和 `llm_rag_raw_fewshot.py` 的每条 JSONL record 中增加：

```json
"prompt_chars": 12345,
"prompt_bytes": 23456,
"active_feature_count": 37
```

实现要求：

1. `prompt_chars = len(prompt)`
2. `prompt_bytes = len(prompt.encode("utf-8"))`
3. `active_feature_count` 用当前 test row 中值为 1 的 feature 数，不包含 `class`

### 4.2 校验 evidence 是否来自当前样本

在 record 中增加两类独立字段，**严格区分两种非法 evidence**：

```json
"invalid_evidence_vocab": ["..."],
"invalid_evidence_vocab_count": 0,
"invalid_evidence_inactive": ["..."],
"invalid_evidence_inactive_count": 0
```

判定规则：

1. 设 `drebin_vocab` 为 Drebin 215 个 feature 全集。
2. 设 `active_set` 为当前测试样本中值为 1 的 feature 集合。
3. 遍历 LLM 返回 `evidence` 列表中的每个字符串 `e`（做大小写不敏感和前缀归一化）：
   - 如果 `e ∉ drebin_vocab`：写入 `invalid_evidence_vocab`（典型如 `android.widget.Button`，是模型彻底幻觉）。
   - 如果 `e ∈ drebin_vocab` 但 `e ∉ active_set`：写入 `invalid_evidence_inactive`（即“数据集里有这个特征，但本样本没激活，模型在编造证据”）。
4. 两类都要计数，互不重叠。

metrics 增加：

```json
"invalid_evidence_vocab_total": 0,
"invalid_evidence_vocab_rate": 0.0,
"invalid_evidence_inactive_total": 0,
"invalid_evidence_inactive_rate": 0.0
```

诊断含义：

1. `invalid_evidence_vocab_rate` 高 → 模型幻觉严重，prompt 被截断或 prompt 没有把 Drebin vocab 限制清楚。
2. `invalid_evidence_inactive_rate` 高 → 模型理解了 Drebin 但伪造证据，常出现在 semantic 描述太“危险化”的情况下。

验收目标：`semantic-neutral-fixed` 的两个 rate 都应明显低于 `semantic-risky-old`，且 `invalid_evidence_vocab_rate` 接近 0。

5. 不使用 `data/processed/fewshot_seed42_test100/test.csv` 的标签做任何统计。

输出文件：

```text

data/processed/feature_stats/seed42_test100/feature_stats_metadata.json
```

`feature_stats.csv` 字段：

```text
feature
category
support_B
support_S
pool_B_total
pool_S_total
p_feature_given_B
p_feature_given_S
log_odds_S_vs_B
abs_log_odds
stat_direction
```

`stat_direction` 取值：

```text
leans_B
leans_S
weak_or_mixed
```

建议阈值：

```python
if log_odds_S_vs_B >= 0.5:
    stat_direction = "leans_S"
elif log_odds_S_vs_B <= -0.5:
    stat_direction = "leans_B"
else:
    stat_direction = "weak_or_mixed"
```

log-odds 使用 Laplace smoothing，避免 0 概率：

```python
p_B = (support_B + 1) / (pool_B_total + 2)
p_S = (support_S + 1) / (pool_S_total + 2)
log_odds_S_vs_B = log(p_S / (1 - p_S)) - log(p_B / (1 - p_B))
```

### 5.2 训练池统计在 prompt 中的使用方式

训练池统计是数据事实，不是标签泄漏。它可以出现在：

1. `semantic-neutral-fixed` 的每个 active feature 描述中。
2. `feature_card` 的 `prompt_text` 中。
3. `behavior_rule` 的解释中。

推荐措辞：

```text
Training-pool statistic: this feature leans_B / leans_S / weak_or_mixed.
Use this only as dataset context, not as a direct label.
```

不要写成：

```text
This feature proves the app is benign/malware.
```

**报告写作注（实验诚信声明）**：`P(feature|B)`、`P(feature|S)`、`log_odds_S_vs_B` 在数学形式上等同于朴素贝叶斯的类条件概率，相当于把训练池的边缘先验显式喂给 LLM。这与“在 prompt 中加方向性引导词”有本质区别：前者注入的是从训练池（排除 test）计算得到的客观数据分布，未引入人类对“是否应该判 S”的主观偏好。最终报告里要在“方法”部分明确这一点，避免被评阅人误读为数据泄漏。验证手段是检查 `feature_stats_metadata.json` 中 `test_indices_excluded` 字段，并和 `test_metadata.json` 中的 `test_source_indices` 严格相等。

### 6.2 `paper_prompt_utils.py` 渲染规则

`semantic-risky-old`：

1. 使用旧的 `feature_semantics_gemma.json`（不修改、不替换）。
2. 保持旧行为，用作对照和错误分析。

`semantic-neutral-fixed`：

每个 active feature 渲染为结构化中性描述：

```text
- feature: Ljavax.crypto.Cipher
  category: API call signature
  meaning: Java cryptographic cipher API used for encryption/decryption operations.
  train_stat: leans_B, P(feature|B)=0.536, P(feature|S)=0.264
```

要求：

1. `meaning` 只写字面含义，不写“窃取”“恶意 payload”“隐藏真实逻辑”等强结论。
2. 如果某个 feature 命中 behavior rule，强结论只能来自 rule block，而不是单个 feature meaning。
3. 对 `leans_B` 的特征，不允许在单特征描述中渲染成恶意倾向。

**实现细节：新增 `row_to_semantic_neutral_text()` 函数与 dispatcher**

注意当前 `fewshot_utils.row_to_feature_text()` 的实际签名是：

```python
def row_to_feature_text(
    row: pd.Series,
    feature_categories: dict[str, str] | None = None,
    label_col: str = "class",
) -> str:
```

它**没有** `max_features` 参数。为保持公平比较（raw / semantic-risky-old / semantic-neutral-fixed 三组的 active feature 数必须一致），下面所有 3 个 render 函数也都不引入 `max_features`，全部一律渲染当前样本的全部 active features。如果将来需要做长度消融，再在 `fewshot_utils.py` 集中扩展一次签名，三处同步修改。

新增函数签名（`paper_prompt_utils.py`）：

```python
def row_to_semantic_neutral_text(
    row: pd.Series,
    feature_categories: dict[str, str],
    semantics_neutral: dict[str, dict],
    feature_stats: dict[str, dict],
    label_col: str = "class",
) -> str:
    """渲染 semantic-neutral-fixed 的 active feature 描述块。

    Args:
        row: 当前测试样本的一行数据，列名为 Drebin feature。
        feature_categories: feature -> category（如 "API call signature"）。
        semantics_neutral: feature -> {"meaning": str, ...}，
            来源于 feature_semantics_neutral_stats.json。
        feature_stats: feature -> {"stat_direction": str,
            "p_feature_given_B": float, "p_feature_given_S": float, ...}，
            来源于 feature_stats.csv（已转为 dict）。
        label_col: 与 row_to_feature_text 保持一致，过滤标签列。

    Returns:
        多行字符串，每个 active feature 一个 4 字段 block。
        缺失字段统一填 "unknown" 或保留为缺省，不引入 fallback 风险词。
    """
```

字段缺失处理：

1. `semantics_neutral` 中查不到 → `meaning: "unknown"`。
2. `feature_stats` 中查不到 → `train_stat: "weak_or_mixed, P(feature|B)=NA, P(feature|S)=NA"`。
3. `feature_categories` 中查不到 → `category: "unknown"`。
4. 不允许用 “may be used for malicious purpose” 等任何带方向的兜底文案。

dispatcher 改写（`paper_prompt_utils.py` 的 `row_to_feature_expr_text()`）：

```python
def row_to_feature_expr_text(
    row: pd.Series,
    feature_expr: str,
    feature_categories: dict[str, str],
    semantics_risky_old: dict | None = None,
    semantics_neutral: dict | None = None,
    feature_stats: dict | None = None,
    label_col: str = "class",
) -> str:
    if feature_expr == "raw":
        return row_to_feature_text(row, feature_categories, label_col=label_col)
    if feature_expr in {"semantic", "semantic-risky-old"}:
        # 旧别名 "semantic" 等价于 "semantic-risky-old"
        return row_to_semantic_feature_text(
            row, feature_categories, semantics_risky_old, label_col=label_col,
        )
    if feature_expr == "semantic-neutral-fixed":
        return row_to_semantic_neutral_text(
            row, feature_categories, semantics_neutral, feature_stats,
            label_col=label_col,
        )
    raise ValueError(f"unknown feature_expr: {feature_expr}")
```

如果当前 `row_to_semantic_feature_text()` 现行签名没有 `label_col` 参数，coding agent 需要在此函数也补上该参数并把过滤逻辑对齐到 `row_to_feature_text`，三个 render 函数对 `label_col` 的行为必须完全一致。

`llm_feature_expr_fewshot.py` 在 `--feature-expr all` 模式下，依次对 `raw`、`semantic-risky-old`、`semantic-neutral-fixed` 各跑一次完整循环，输出 3 套独立 jsonl 与 metrics。

### 6.3 生成 neutral semantic 文件（方案三：用全新中性 prompt 让 Gemma 重生成）

目标文件：`generate_feature_semantics.py`

本节明确采用**方案三**：不直接复用 `feature_semantics_gemma.json` 的旧 `meaning`，而是使用一份全新的、强约束的中性 prompt，调用 `gemma4:e4b` 重新生成 215 条 meaning，再合并训练池统计字段，输出 `feature_semantics_neutral_stats.json`。

理由：旧 JSON 中很多 meaning 含有 “execute external code / hidden logic / steal / exfiltrate / suspicious” 之类的风险词，这些词正是导致 semantic 偏 S 的核心来源；只用模板覆盖大类则缺乏 API 级精度。重新生成既保证中性，又保留 215 条特征的差异化描述。

新增生成模式：

```text
--style neutral-fixed
--feature-stats data/processed/feature_stats/seed42_test100/feature_stats.csv
--output-json data/processed/paper_features/feature_semantics_neutral_stats.json
--model gemma4:e4b
--temperature 0.0
--request-timeout 120
--num-ctx 4096
--max-feature-batch 1
```

**生成流程**：

1. 遍历 `data/dataset-features-categories.csv` 中 215 个 feature。
2. 对每个 feature 调用一次 Gemma，使用下面 §6.3.1 的固定 prompt 模板。
3. 解析返回 JSON，取 `meaning` 字段。
4. 与 `feature_stats.csv` 对应行合并 `stat_direction` / `p_feature_given_B` / `p_feature_given_S`。
5. 写入 `feature_semantics_neutral_stats.json`。
6. 失败重试 2 次；仍失败则写 `"meaning": "unknown (generation failed)"`，记录到 `generation_failures.json`，**不允许**用风险词兜底文案。

#### 6.3.1 全新中性 prompt 模板

固定模板（写入 `generate_feature_semantics.py` 内部常量 `NEUTRAL_PROMPT_TEMPLATE`）：

```text
You are documenting Android API and permission features for a research dataset.

TASK
Write a single short sentence describing what the following Android feature literally is or does.

CONSTRAINTS
- Only describe the literal Android documentation meaning of the feature.
- Do not say whether the feature is malicious, suspicious, dangerous, harmful, risky, or benign.
- Do not say what kind of app uses it, what the attacker can do with it, or what the user should worry about.
- Do not use any of these words: malicious, malware, suspicious, harmful, dangerous, risky, attack, exploit, exfiltrate, steal, hidden, secret, payload, victim, abuse, hack.
- Do not write more than 25 words.
- Output strict JSON only, no surrounding text.

INPUT
feature_name: {feature_name}
feature_category: {feature_category}

OUTPUT JSON SCHEMA
{
  "feature": "<echo feature_name>",
  "meaning": "<one neutral sentence, <=25 words>"
}
```

`{feature_category}` 取自 `data/dataset-features-categories.csv` 中的 category 列（如 `API call signature`、`Manifest Permission`、`Intent`、`Commands signature` 等）。

**Prompt 设计说明（不是引导词）**：

1. 上面 prompt 中给出的是“风格约束”（中性、字面、短句、禁用情感色彩词），不是“判 B / 判 S”的方向性引导。
2. 这些约束作用在**特征字典生成阶段**，不进入下游 `llm_feature_expr_fewshot.py` 的分类 prompt。
3. 分类 prompt 看到的只是 25 词内的中性 meaning + 训练池统计字段，与 raw 模式公平比较。

#### 6.3.2 验证

生成后必须执行：

1. 用正则扫描 `meaning` 字段，确认 215 条全部不含禁用词列表（malicious/malware/suspicious/.../hack）。命中条数必须 = 0；命中则该条标记 `regenerate` 并重跑直到通过。
2. 用 `len(meaning.split())` 统计，95 分位数应 <= 25。
3. 抽样 20 条人工 spot check（推荐 1 个 B-leaning、1 个 S-leaning、1 个 weak/mixed，各类各 ~6 条）。

输出 JSON 每个 feature 至少包含：

```json
{
  "feature": "Ljavax.crypto.Cipher",
  "category": "API call signature",
  "meaning": "Java standard library class providing symmetric and asymmetric cipher operations such as encryption and decryption.",
  "stat_direction": "leans_B",
  "p_feature_given_B": 0.536,
  "p_feature_given_S": 0.264
}
```

不强制限定写法风格只用模板；重点是**中性、短、稳定、可复现**。`temperature=0.0` + `num_ctx=4096` 是为了让 215 次生成尽量稳定，多次重跑结果应高度一致。生成耗时预算：~5-10 分钟（单机本地 Ollama）。

### 8.1 `rag_retriever.py` 新增方法（方案二：重建 FAISS 子索引并持久化）

本节明确采用**方案二**：在构建 KB 时一次性把整个语料按 `doc_type` 拆成两个独立 FAISS 索引并写盘，运行时按需加载，避免每次推理都对全量 embedding 做掩码筛选。

#### 8.1.1 KB 构建阶段：拆出两个 FAISS 子索引

修改 `rag_kb_builder.py`。**沿用现有项目命名风格**（实际目录里是 `kb.index` / `kb_docs.jsonl` / `kb_embeddings.npy` / `kb_metadata.json`，不是 `kb_index.faiss`），不要另起一套命名：

2. 额外构建：

```json

    "size": 215
  },

  }
}
```

#### 8.1.2 `rag_retriever.py` 加载阶段

**复用现有 `RetrievalHit` dataclass**（`rag_retriever.py:52` 定义为 `@dataclass(frozen=True)`，字段为 `rank: int / doc_index: int / score: float / doc: dict`）。不要新增 `doc_id` / `doc_type` 字段，`doc_type` 通过 `hit.doc["doc_type"]` 取，`doc_id` 通过 `hit.doc["doc_id"]`（或更稳妥地 `hit.doc.get("doc_id")`）表达。

```python

# 子索引内行号 i → kb_docs.jsonl 中的全局 doc_index

```

`__init__` 中根据 `kb_metadata.json` 的 `sub_indices` 段，按 `faiss.read_index` 加载两个子索引、按 `json.load` 加载两个 doc_index map 到字典中；如果文件缺失则在 `bucketed` 模式时 raise FileNotFoundError，**不允许**回退到“在全量 index 上掩码筛选”的实现（性能与可复现性都更差）。

#### 8.1.3 新增检索方法

由于 `RetrievalHit` 是 `frozen=True`，**不能**在合并后用 `hit.rank = new_rank` 重写字段。必须用 `dataclasses.replace()` 重建新实例，或在合并时按顺序新构造。

```python
from dataclasses import replace

def retrieve_by_doc_type(
    self,
    query_text: str,
    doc_type: str,
    top_k: int,
) -> list[RetrievalHit]:
    """在指定 doc_type 的独立 FAISS 子索引上做 top-k 检索。

    命中后用子索引行号查 `sub_doc_index_maps[doc_type]` 得到 kb_docs.jsonl 中的
    全局 doc_index，从而构造与现有 retrieve() 完全兼容的 RetrievalHit。
    """
    index = self.sub_indices[doc_type]
    index_map = self.sub_doc_index_maps[doc_type]
    q_emb = encode_texts(
        self.model,
        [query_text],
        batch_size=self.batch_size,
        show_progress_bar=False,
    )  # shape (1, dim), L2-normalized
    scores, sub_idxs = index.search(q_emb, top_k)
    hits: list[RetrievalHit] = []
    for rank, (sub_i, score) in enumerate(zip(sub_idxs[0], scores[0]), start=1):
        if sub_i < 0:
            continue
        global_doc_index = index_map[int(sub_i)]
        hits.append(
            RetrievalHit(
                rank=rank,
                doc_index=global_doc_index,
                score=float(score),
                doc=self.docs[global_doc_index],
            )
        )
    return hits


def retrieve_bucketed(
    self,
    feature_query_text: str,
    rule_query_text: str,
    feature_top_k: int = 2,
    rule_top_k: int = 3,
) -> list[RetrievalHit]:
    """先在 feature_card 子索引检索 feature_top_k 条，再在 behavior_rule 子索引检索 rule_top_k 条。"""
    feat_hits = self.retrieve_by_doc_type(feature_query_text, "feature_card", feature_top_k)
    rule_hits = self.retrieve_by_doc_type(rule_query_text, "behavior_rule", rule_top_k)
    combined: list[RetrievalHit] = []
    for new_rank, hit in enumerate(feat_hits + rule_hits, start=1):
        # 由于 RetrievalHit 是 frozen dataclass，必须用 replace 重建
        combined.append(replace(hit, rank=new_rank))
    return combined
```

返回顺序：先 feature_card hits，再 behavior_rule hits；`rank` 在合并后从 1 连续编号。

下游消费方拿到 `hits` 后，取 `hit.doc["doc_type"]` 区分桶（`feature_card` / `behavior_rule`），取 `hit.doc["doc_id"]` 作 `retrieval_logs.jsonl` 中的 `doc_id` 字段（对应 §10 中 `behavior_rule_hit_rate` 计算公式里 `h["doc_type"]` 和 `h["doc_id"]` 的来源）。

**`behavior_rule_hit_rate` 计算公式**（只对 RAG 实验有意义；其他方法填 `N/A`）：

设 `N` 为本次实验 test_size，记每个样本 `i` 的 retrieval hits 列表为 `H_i`，定义：

```python
def behavior_rule_hit_rate(retrieval_logs: list[dict]) -> float:
    """Fraction of test samples whose retrieval result contains >= 1 behavior_rule hit.

    Lower-bounded by validation target 0.95 in smoke test.
    """
    n_total = len(retrieval_logs)
    if n_total == 0:
        return 0.0
    n_hit = sum(
        1
        for sample in retrieval_logs
        if any(h["doc_type"] == "behavior_rule" for h in sample["hits"])
    )
    return n_hit / n_total
```

`retrieval_logs.jsonl` 每行一条样本，结构：

```json
{
  "test_index": 10863,
  "hits": [
    {"doc_id": "FEAT_INTERNET", "doc_type": "feature_card", "rank": 1, "score": 0.81},
    {"doc_id": "FEAT_LJAVAX_CRYPTO_CIPHER", "doc_type": "feature_card", "rank": 2, "score": 0.74},
    {"doc_id": "RULE_SMS_ABUSE", "doc_type": "behavior_rule", "rank": 3, "score": 0.69},
    {"doc_id": "RULE_DEVICE_ID_EXFIL", "doc_type": "behavior_rule", "rank": 4, "score": 0.65},
    {"doc_id": "RULE_STANDARD_IPC", "doc_type": "behavior_rule", "rank": 5, "score": 0.61}
  ]
}
```

辅助指标（可选写入 metrics，便于错误分析）：

```python
avg_behavior_rule_hits_per_sample = mean([
    sum(1 for h in s["hits"] if h["doc_type"] == "behavior_rule")
    for s in retrieval_logs
])
behavior_rule_label_distribution = Counter(
    h.get("related_label", "unknown")
    for s in retrieval_logs
    for h in s["hits"] if h["doc_type"] == "behavior_rule"
)
```

为避免 `behavior_rule_label_distribution` 全部落到 `unknown`，需要同步修改 `RetrievalHit.to_log_dict()`：额外写入 `related_label: self.doc.get("related_label")`。

## 14. 禁止事项

1. 不要在 prompt 中加入“如果不确定就判 B”“不要轻易判 S”“倾向选择 B”等任何方向性引导词。
2. 不要后处理把低置信度 `S` 改成 `B`，也不要反向操作。
3. 不要使用 test 标签计算 feature statistics；`feature_stats.py` 必须从全量 Drebin 中剔除 `test_metadata.json` 的 `test_source_indices` 后再算。
4. 不要把 `related_label` 写成“命中即输出该标签”的硬规则；它在 prompt 中只能以 soft prior 出现。
5. 不要覆盖 `results/claude_audit_30/`、`results/codex_audit/` 和旧 `results/feature_expr_llm/` 结果；新的 smoke 写入 `results/fix_plan_v1_smoke/`，新的正式实验写入 `results/final/`。
6. 不要只报告 accuracy，必须报告 `recall_benign`、`recall_malware`、`macro_f1`、`pred_S_ratio`、两个 `invalid_evidence_*_rate` 和 confusion matrix。
7. 不要让 `semantic-risky-old` 静默继承 `semantic-neutral-fixed` 的中性描述；前者必须严格使用旧 `feature_semantics_gemma.json`，否则失去对照价值。
8. 不要在 `generate_feature_semantics.py --style neutral-fixed` 失败时回退到“风险词兜底文案”；只允许写 `"unknown (generation failed)"`，并加入 `generation_failures.json` 等待 retry。
