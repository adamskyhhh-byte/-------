# 少样本实验数据划分与实验改造计划

## 1. 当前方案理解

目前项目需要从原来的“80% 训练、20% 测试”传统机器学习基线，改造成一个统一的 few-shot 少样本实验流程。

新的核心思路是：

1. 原始 `data/` 数据保持不变。
2. 单独编写一个数据划分脚本，从原始数据中复制出一份实验用数据。
3. 划分结果包含两部分：
   - 学习数据：随机选出的少量样本，0 类和 1 类数量一致。
   - 验证数据：剩余样本或从剩余样本中再抽取出的测试样本。
4. 后续所有实验都基于这份固定划分：
   - 传统机器学习 baseline。
   - LLM-only。
   - LLM + RAG。
5. 学习数据数量由运行 Python 脚本时传入的参数决定。

这个方向是合理的，因为它能保证不同方法在相同数据条件下比较，避免传统 ML 使用大量训练数据而 LLM 只使用 few-shot 样本导致的不公平对比。

## 2. 推荐的整体流程

### 2.1 数据划分脚本

建议新增一个独立脚本，例如：

```text
prepare_fewshot_split.py
```

脚本职责只做数据准备，不训练模型、不调用 LLM。

输入：

```text
data/drebin-215-dataset-5560malware-9476-benign.csv
data/dataset-features-categories.csv
```

输出建议放在：

```text
data/processed/fewshot_k{K}_seed{SEED}/
```

目录内建议包含：

```text
train.csv        # few-shot 学习数据
test.csv         # 验证/测试数据
metadata.json    # 本次划分的参数、样本数量、标签分布、随机种子
```

命令示例：

```bash
python prepare_fewshot_split.py --k 5 --seed 42 --test-per-class 100
```

其中 `--k 5` 建议表示“每个类别选 5 个样本”，所以学习集总数是 10 行。

### 2.2 标签处理

Drebin 原始标签是：

```text
B = Benign / 良性
S = Malware / 恶意
```

现有 baseline 中可能映射成：

```text
B -> 0
S -> 1
```

建议在保存的 `train.csv` 和 `test.csv` 中保留原始 `class` 列，不直接覆盖成 0/1。各实验脚本内部再根据需要做映射。

这样做的好处是：

1. LLM prompt 可以直接使用 `B` / `S`。
2. 传统 ML 可以自己映射为 0/1。
3. 避免后续不同脚本对标签含义产生歧义。

## 3. 数据划分规则

### 3.1 学习数据

从每个类别随机抽取 K 个样本。

例如 K=5：

```text
良性 B: 5 个
恶意 S: 5 个
总计: 10 个
```

这些样本用于：

1. 传统 ML 的训练集。
2. LLM-only 的 few-shot examples。
3. LLM + RAG 的 few-shot examples 或上下文示例。

### 3.2 验证数据

有两种可选方案。

方案 A：使用除学习数据以外的全部剩余数据作为验证集。

优点：

1. 测试样本多，统计更稳定。
2. 接近完整数据集评估。

缺点：

1. 数据集本身类别不平衡，良性样本更多，accuracy 可能虚高。
2. LLM 推理成本很高，几千上万个样本会很慢。
3. 不同 K 值会导致剩余验证集略有变化。

方案 B：从剩余数据中再平衡抽取固定数量作为验证集。

例如：

```text
良性 B: 100 个
恶意 S: 100 个
总计: 200 个
```

优点：

1. 类别平衡，指标更公平。
2. LLM 推理时间可控。
3. 更适合报告中的方法对比。

缺点：

1. 测试集样本少于全部剩余数据。
2. 需要记录测试集抽样随机种子，保证可复现。

当前更推荐方案 B，尤其适合 LLM-only 和 LLM + RAG。

## 4. Baseline 改造计划

当前 `baseline-train.ipynb` 的主要问题是使用了：

```python
train_test_split(..., test_size=0.2)
```

这会让传统机器学习模型使用大量训练样本，和 few-shot LLM 不公平。

改造后 baseline 应该：

1. 不再调用 `train_test_split`。
2. 从数据划分脚本生成的目录读取：

```text
train.csv
test.csv
```

3. 使用 `train.csv` 训练传统 ML 模型。
4. 使用 `test.csv` 评估模型。
5. 输出 accuracy、precision、recall、F1、confusion matrix。
6. 在结果中明确标注：

```text
K = 5
seed = 42
训练样本 = 每类 5 个，共 10 个
测试样本 = 每类 100 个，共 200 个
```

原来的 80/20 baseline 可以保留，但建议只作为“数据充足时的传统 ML 上限参考”，不参与主实验公平对比。

## 5. LLM-only 改造计划

LLM-only 实验应该读取同一份：

```text
train.csv
test.csv
```

执行流程：

1. 将 `train.csv` 中的样本转换为 few-shot examples。
2. 每次预测 `test.csv` 中的一个样本时，把 few-shot examples 放入 prompt。
3. 调用 Ollama 的 `gemma4:e4b`。
4. 要求模型输出严格 JSON。
5. 保存每个测试样本的预测结果到 JSONL。

建议输出路径：

```text
results/predictions/llm_only_k{K}_seed{SEED}.jsonl
```

每条记录建议包含：

```json
{
  "idx": 0,
  "true_label": "S",
  "pred_label": "S",
  "parse_ok": true,
  "evidence": ["SEND_SMS", "READ_PHONE_STATE"],
  "explanation": "模型给出的解释",
  "raw": "LLM 原始输出"
}
```

## 6. LLM + RAG 后续衔接

LLM + RAG 也应该使用同一份数据划分。

区别只在 prompt 构造阶段：

1. 先把测试样本转换成特征文本。
2. 用特征文本从规则库中检索 Top-K 条相关规则。
3. 将 few-shot examples、样本特征、检索到的规则一起放入 prompt。
4. 要求输出标签、证据、引用规则、解释和置信度。

这样可以保证和 LLM-only 的唯一区别是“是否引入规则/RAG”，便于做消融实验。

## 7. 推荐的实验对比表

主实验建议先固定：

```text
K = 5
seed = 42
test_per_class = 100
```

对比：

| 方法 | 训练/示例数据 | 测试数据 | 目的 |
|---|---:|---:|---|
| 传统 ML baseline | 每类 K 个 | 固定测试集 | 少样本传统模型表现 |
| LLM-only | 每类 K 个 few-shot examples | 固定测试集 | 纯 LLM 少样本能力 |
| LLM + RAG | 每类 K 个 + 检索规则 | 固定测试集 | 规则增强是否有效 |

进阶实验可以跑：

```text
K = 1, 3, 5, 10
```

画 K vs Accuracy / Macro-F1 曲线。

## 8. 需要决策的问题

### 8.1 参数 K 的含义

需要确认：

```text
--k 5
```

是表示“每类 5 个”，还是表示“总共 5 个”。

当前建议：表示每类 5 个。

### 8.2 验证集使用全部剩余数据，还是平衡抽样

需要确认：

1. 使用全部剩余数据。
2. 从剩余数据中每类抽取固定数量，比如每类 100 个。

当前建议：每类抽取固定数量，默认每类 100 个。

### 8.3 不同 K 实验是否共用同一个测试集

如果未来要画 K vs Accuracy 曲线，需要决定不同 K 是否使用完全相同的测试集。

推荐做法：

1. 先固定抽取一个测试集。
2. 再从测试集之外的数据里抽取 K-shot 学习数据。

这样 K=1、K=3、K=5、K=10 的测试集完全一致，曲线更公平。

但这和“先抽学习数据，再把剩下的作为验证数据”的流程略有不同，需要你确认最终采用哪种。

### 8.4 标签保存格式

需要确认保存文件中标签是否：

1. 保留原始 `B` / `S`。
2. 转成 `0` / `1`。
3. 两者都保存。

当前建议：保留原始 `class`，必要时额外增加 `label_id`，但不要删除原始标签。

### 8.5 是否需要多随机种子

few-shot 的结果会受具体抽到哪几个样本影响。

需要确认是否只跑：

```text
seed = 42
```

还是后续跑多个 seed：

```text
seed = 42, 0, 1
```

当前建议：先实现单 seed，报告阶段如果时间允许再补多 seed 均值和方差。

### 8.6 Core-set 是否现在一起设计

PPT 中 Core-set 是进阶项。

需要确认当前数据划分脚本是否只做 random k-shot，还是预留：

```bash
--strategy random
--strategy coreset
```

当前建议：脚本接口先预留 `--strategy random`，但第一版只实现 random，避免过早复杂化。

## 9. 建议的实现顺序

1. 新增数据划分脚本，生成 `train.csv`、`test.csv`、`metadata.json`。
2. 先用 K=5、seed=42 生成一份固定实验数据。
3. 修改 baseline，使其从固定划分读取训练集和测试集。
4. 编写 LLM-only 脚本，使其从同一份固定划分读取数据。
5. 跑通传统 ML baseline 和 LLM-only 的 K=5 对比。
6. 再实现 LLM + RAG。
7. 最后补 K 曲线、多 seed 或 Core-set。

## 10. 本阶段不做的事情

本计划阶段暂时不做：

1. 不修改原始 `data/` 文件。
2. 不直接改动现有 baseline 代码。
3. 不开始写 LLM-only 完整实验脚本。
4. 不构建 RAG 规则库。
5. 不跑耗时 LLM 批量推理。

等上述不确定问题确认后，再进入代码实现阶段。
