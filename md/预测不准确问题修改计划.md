# 预测不准确问题修改计划

> 生成时间：2026-05-20  
> 依据材料：`md/项目方案.md`、`6.experiment-20260508.pptx`、`md/Claude对预测不准确问题生成的审查报告.md`、`md/Codex对预测不准确问题生成的审查报告.md`  
> 目标实验：`raw-data-LLM-only`、`semantic-data-LLM-only`、`raw-data-LLM+RAG`，并为 PPT 要求中的传统 ML/NN baseline 与 Core-set/RAG 扩展预留位置。

## 1. 总体判断

两份审查报告的结论基本一致：当前预测不准的主要形态不是随机错误，而是模型在 `semantic-data-LLM-only` 和 `raw-data-LLM+RAG` 中明显过度预测 `S`。`raw-data-LLM-only` 相对更稳，可以先作为参照组保留。

关键根因分四层：

| 层级 | 问题 | 影响 |
|---|---|---|
| 数据分布 | Drebin 中不少良性样本 active feature 更多，且 `Cipher`、`Binder`、反射、`mount` 等特征在 B 类中并不少见 | LLM 的“特征多/敏感 API 多就可疑”先验与数据事实冲突 |
| 语义化表达 | semantic prompt 展开后过长，且描述里风险词密度高 | 上下文截断、证据幻觉、把中性特征解释成恶意线索 |
| RAG 检索 | 215 条 `feature_card` 与 16 条 `behavior_rule` 混在同一个 top-k 池里竞争 | 检索几乎只命中特征卡，规则增强失效，还强化偏 S |
| Prompt/调用 | 部分脚本仍把 `S` 定义为 `malware / malicious / suspicious`，Ollama 默认上下文不足 | 判 S 阈值被拉低，semantic 输出格式不稳定 |

因此，修改原则是：修输入表示、检索结构、知识库和实验评估，不用“请不要轻易判 S”“不确定就判 B”这类人为引导词。

## 2. 修改目标

1. 让三组 LLM 实验公平可比：同一 split、同一 k、同一模型、同一温度、同一 JSON schema。
2. 让 semantic 实验先恢复有效格式：`strict_parse_ok_rate` 接近 1.0，`evidence` 必须来自当前样本真实 Drebin feature。
3. 让 RAG 真正检索到行为规则：每条样本至少注入若干条 `behavior_rule`，而不是只注入 `feature_card`。
4. 降低误报：重点观察 `recall_benign`、`FP`、`pred_S_ratio`，不能只看 accuracy。
5. 对齐 PPT 评分项：规则库、Prompt、结果表、图表、错误案例分析都要能落到最终报告。

## 3. P0 必改项

### 3.1 给 Ollama 调用增加 `num_ctx`

目标文件：

| 文件 | 修改点 |
|---|---|
| `llm_only_fewshot.py` | `call_ollama()` 增加 `num_ctx` 参数，并写入 `options` |
| `llm_feature_expr_fewshot.py` | CLI 增加 `--num-ctx`，调用 `call_ollama()` 时传入 |
| `llm_rag_raw_fewshot.py` | CLI 增加 `--num-ctx`，调用 `call_ollama()` 时传入 |

建议默认：

```text
--num-ctx 12288
```

如果显存或内存吃紧，降为：

```text
--num-ctx 8192
```

验收标准：

| 指标 | 目标 |
|---|---:|
| semantic `strict_parse_ok_rate` | >= 0.90 |
| semantic `parse_ok_rate` | >= 0.95 |
| semantic evidence 非 Drebin 特征名比例 | 接近 0 |

### 3.2 清理 `suspicious` 标签映射

目标文件：

| 文件 | 当前问题 | 修改 |
|---|---|---|
| `llm_only_fewshot.py` | prompt 里 `S means malware / malicious / suspicious` | 改成 `S means malware app` |
| `llm_rag_raw_fewshot.py` | RAG prompt 同样把 suspicious 映射为 S | 改成与 `llm_feature_expr_fewshot.py` 一致 |

注意：这不是加入“偏 B”提示，而是移除原有的低阈值词。`llm_feature_expr_fewshot.py` 当前已经比较干净，可以作为统一模板。

验收标准：

| 指标 | 目标 |
|---|---:|
| raw/RAG prompt 中 `suspicious` 出现次数 | 0 |
| 三组 prompt 的 label mapping | 完全一致 |

### 3.3 增加 prompt 长度和证据合法性日志

目标文件：

| 文件 | 修改点 |
|---|---|
| `llm_feature_expr_fewshot.py` | 每条 record 增加 `prompt_chars`、`prompt_bytes`、`active_feature_count`、`invalid_evidence_count` |
| `llm_rag_raw_fewshot.py` | 同上，额外记录 `retrieved_doc_types` |

目的：

1. 证明 semantic 是否还会超上下文。
2. 证明 LLM 输出 evidence 是否来自当前样本。
3. 给最终报告的错误案例分析提供材料。

## 4. P1 核心改造项

### 4.1 RAG 改为分桶检索

目标文件：`rag_retriever.py`、`llm_rag_raw_fewshot.py`

当前：

```text
feature_card + behavior_rule 混合进入同一个 FAISS top-k
```

修改为：

```text
feature_card 检索：解释当前 active feature 的含义
behavior_rule 检索：提供行为组合和安全规则依据
最终 prompt = feature hits + rule hits
```

建议 CLI：

```text
--feature-top-k 2
--rule-top-k 3
--retrieval-mode bucketed
```

实现建议：

1. 在 `RagRetriever` 中加载 `kb_embeddings.npy`。
2. 新增 `retrieve_by_doc_type(query_text, doc_type, top_k)`。
3. 对指定 `doc_type` 的 embedding 子矩阵做相似度排序，避免全局 top-k 把规则挤掉。
4. 保留旧的 `retrieve()` 作为 `global` 模式，方便做 ablation。

验收标准：

| 指标 | 目标 |
|---|---:|
| RAG 每条样本命中 `behavior_rule` 数 | >= 2 |
| RAG retrieval log 中 `behavior_rule_hit_rate` | 接近 1.0 |
| RAG 不再出现 30/30 全预测 S 的塌缩 | 必须满足 |

### 4.2 RAG query 拆成 feature query 和 rule query

目标文件：`llm_rag_raw_fewshot.py`

建议新增两个 query 构造函数：

| 函数 | 用途 |
|---|---|
| `build_feature_query_text()` | 包含 active raw feature names 和 category，用于找单特征解释 |
| `build_rule_query_text()` | 包含高危组计数、关键组合、类别覆盖，用于找行为规则 |

规则 query 不需要列出全部 inactive feature，但应给出 compact group count：

```text
SMS group active count
root/system-command group active count
package-install group active count
telephony-identifier group active count
dynamic-loading group active count
network group active count
```

目的：让模型看到“哪些高危组合没有出现”，避免只看见 `Cipher`、`Binder`、反射就判 S。

### 4.3 改造知识库：规则需要更平衡

目标文件：`rag_kb_builder.py`

当前知识库里 `feature_card` 占 93% 以上，`behavior_rule` 数量少，而且大多是风险解释。建议新增或调整：

| 类型 | 数量建议 | 例子 |
|---|---:|---|
| 恶意风险规则 | 10-15 | `SEND_SMS + READ_SMS + INTERNET` |
| 良性常见规则 | 6-10 | `bindService + onServiceConnected + Binder` 属于常见 IPC |
| 上下文依赖规则 | 6-10 | `Cipher`、`DexClassLoader`、`mount` 需要结合网络、短信、root 等判断 |

每条规则建议包含字段：

```text
doc_id
doc_type
category
feature_pattern
related_label
retrieval_text
prompt_text
source
leakage_risk
```

其中 `related_label` 可以使用：

```text
S / B / Context-dependent
```

这更符合 PPT 中“Feature | Related Label | Explanation”的规则库要求。

### 4.4 给知识库加入训练集统计

新增或修改目标：

| 文件 | 作用 |
|---|---|
| 新增 `feature_stats.py` 或集成进 `rag_kb_builder.py` | 计算每个 feature 的 `P(feature|B)`、`P(feature|S)`、log-odds |
| `rag_kb_builder.py` | 把统计摘要写进 `feature_card` 和部分 `behavior_rule` |

重要约束：统计不能用共享测试集标签。可用方案是读取 `data/processed/fewshot_seed42_test100/test_metadata.json` 中的 `test_source_indices`，从原始 Drebin 全量数据中排除这些样本，再计算训练池统计。

建议加入的字段：

```text
support_B
support_S
p_feature_given_B
p_feature_given_S
log_odds_S_vs_B
stat_direction
```

这样 `Cipher`、`Binder`、反射等在 B 中更常见的特征，不会继续被知识库自动渲染成恶意证据。

## 5. P2 结构性提升项

### 5.1 semantic 表达降风险词密度

目标文件：`paper_prompt_utils.py`、`generate_feature_semantics.py`

当前 semantic 是：

```text
feature: 中文语义描述
```

建议改成更中性的结构：

```text
- feature: <原始特征名>
  category: <Manifest Permission/API/Intent/Command>
  meaning: <中性含义>
  train_stat: <B 中更常见 / S 中更常见 / 区分度弱>
```

不建议在单个 feature 描述里直接写“窃取”“隐藏真实逻辑”“恶意 payload”等强结论，除非来自匹配到的行为规则或训练统计。

### 5.2 增加数据驱动的 feature subset

目标文件：`paper_prompt_utils.py`、`llm_feature_expr_fewshot.py`

当前 `--feature-subset` 只有 `full`。建议新增：

| subset | 作用 |
|---|---|
| `top_log_odds` | 只保留区分度高的 active features |
| `category_capped` | 每个类别最多保留 N 个，避免长 prompt |
| `rule_matched` | 优先保留命中规则的特征 |

正式三组实验要使用同一种 subset 策略，保证公平。旧的 `full` 可保留做消融。

### 5.3 Core-set 采样

目标文件：`prepare_fewshot_split.py`

PPT 进阶要求比较 Random Sampling 与 Core-set Sampling。建议新增：

```text
--sampling random
--sampling coreset
```

输出目录建议：

```text
data/processed/fewshot_seed42_test100_random/k5/train.csv
data/processed/fewshot_seed42_test100_coreset/k5/train.csv
```

Core-set 可用每类 KMeans 中心最近样本实现。正式报告中至少比较 `k=5` 的 Random vs Core-set；时间允许再跑 `k=1/3/5`。

## 6. 推荐实施顺序

### 阶段 A：冻结现状，保留对照

1. 保留 `results/claude_audit_30/` 和 `results/codex_audit/` 作为问题复现证据。
2. 不覆盖旧结果，新结果统一写到：

```text
results/fix_plan_v1/
```

### 阶段 B：先修 P0

1. 加 `--num-ctx`。
2. 移除 prompt 中的 `suspicious` 映射。
3. 增加 prompt 长度和 evidence 合法性日志。
4. 只跑 30 条 smoke test。

建议命令：

```bash
python llm_feature_expr_fewshot.py --split-root data/processed/fewshot_seed42_test100 --k 5 --feature-expr both --feature-subset full --feature-semantics data/processed/paper_features/feature_semantics_gemma.json --feature-category-path data/dataset-features-categories.csv --output-root results/fix_plan_v1/feature_expr_llm --model gemma4:e4b --temperature 0.1 --request-timeout 180 --num-ctx 12288 --max-test-samples 30
```

### 阶段 C：修 RAG 检索和知识库

1. 实现 bucketed retrieval。
2. 补充良性规则和上下文依赖规则。
3. 加入训练池统计。
4. 先 `--dry-run` 检查 prompt 与 retrieval log。
5. 再跑 30 条 RAG smoke test。

建议命令：

```bash
python llm_rag_raw_fewshot.py --split-root data/processed/fewshot_seed42_test100 --k 5 --feature-top-k 2 --rule-top-k 3 --retrieval-mode bucketed --kb-dir data/processed/rag_kb --feature-category-path data/dataset-features-categories.csv --embedding-model BAAI/bge-small-en-v1.5 --model gemma4:e4b --temperature 0.1 --request-timeout 180 --num-ctx 12288 --output-root results/fix_plan_v1/rag_raw_llm --max-test-samples 30 --local-files-only
```

### 阶段 D：修 semantic 表示

1. 先保留旧 semantic，作为“未校准语义化”的对照。
2. 新增 neutral semantic 或 stats semantic，作为正式 `semantic-data-LLM-only`。
3. 控制 prompt 长度，避免再超过上下文。

### 阶段 E：正式跑实验

建议最小正式矩阵：

| 实验 | k | test size | 输出 |
|---|---:|---:|---|
| 传统 ML baseline | 5 | 200 | `results/final/baseline/` |
| raw-data-LLM-only | 5 | 200 | `results/final/feature_expr_llm/` |
| semantic-data-LLM-only | 5 | 200 | `results/final/feature_expr_llm/` |
| raw-data-LLM+RAG | 5 | 200 | `results/final/rag_raw_llm/` |
| Core-set + RAG | 5 | 200 | `results/final/rag_raw_llm_coreset/` |

建议扩展矩阵：

```text
k = 1 / 3 / 5
sampling = random / coreset
method = raw-only / semantic-only / raw+RAG
```

如果时间不够，优先保证 `k=5, test=200` 完整跑通，再用 `k=1/3` 做 30 条补充分析。

## 7. 评估与报告指标

每组必须报告：

| 指标 | 原因 |
|---|---|
| accuracy | PPT 常规指标 |
| precision | 判断误报影响 |
| recall_malware | 是否漏检恶意 |
| recall_benign | 是否过度误报良性 |
| macro_f1 | 类别不平衡时比 accuracy 更稳 |
| pred_S_ratio | 专门监控偏 S 塌缩 |
| parse_ok_rate / strict_parse_ok_rate | 判断 LLM 输出格式是否可信 |
| invalid_evidence_count | 判断解释是否引用真实特征 |
| behavior_rule_hit_rate | 判断 RAG 是否真的命中规则 |

图表建议：

1. 三组方法 accuracy / macro_f1 对比柱状图。
2. 三组方法 confusion matrix。
3. `pred_S_ratio` 对比图。
4. RAG 检索命中文档类型分布图。
5. 特征 log-odds Top-20 图，解释为什么某些“看似危险”的 API 在 Drebin 中更偏 B。

错误案例分析至少包含：

| 类型 | 数量建议 | 目的 |
|---|---:|---|
| B -> S 误报 | 3 | 说明过度安全先验如何出现 |
| S -> B 漏报 | 1-2 | 如果存在，分析漏检模式 |
| RAG 修复成功案例 | 2 | 证明规则库是否改善解释 |
| RAG 修复失败案例 | 2 | 说明仍然不足的规则或检索问题 |

## 8. 不建议做的事

1. 不要在 prompt 中加入“如果不确定就判 B”。
2. 不要后处理把低置信度 S 改成 B。
3. 不要用测试集标签计算 feature statistics 或调规则。
4. 不要只报告 accuracy，必须同时报告 `recall_benign` 和 `pred_S_ratio`。
5. 不要把 RAG 的失败解释成“LLM 不行”就结束，PPT 明确要求分析规则库和 RAG 是否改善解释。

## 9. 需要你决策的点

### 决策 1：规则库是否显式写 `related_label`

PPT 要求规则格式包含 `Related Label`。当前代码为了避免标签泄漏，规则没有直接写 B/S。



### 决策 2：正式实验是否必须包含传统 ML/NN baseline

PPT 的消融表中方法 1 是传统 ML/NN，且写了“必做：完成方法 1/2/3 的对比”。你的当前重点是三组 LLM/RAG 实验，但仓库里已经有 `baseline_fewshot.py` 和 baseline 结果。



### 决策 3：feature statistics 使用范围

可选方案：

| 方案 | 优点 | 风险 |
|---|---|---|
| 只用 few-shot train.csv | 最严格 few-shot | 样本太少，统计极不稳定 |
| 用排除 test 后的训练池 | 统计稳定，不用测试集标签 | 不再是纯 few-shot 信息 |



### 决策 4：semantic 实验名称是否保留

修完后可能出现两种 semantic：

```text
semantic-risky-old
semantic-neutral-fixed
```



### 决策 5：正式测试规模

可选方案：

| 方案 | 耗时 | 可信度 |
|---|---|---|
| 30 条 | 快 | 只能 smoke test |
| 100 条 | 中等 | 可以初步写报告 |
| 200 条 | 慢 | 与现有 split 完整匹配，最适合作为最终结果 |



### 决策 6：`num_ctx` 取值

建议先试 `12288`。如果 Ollama 显存或内存压力大，再降到 `8192`。

需要你确认：机器是否能接受更慢但更稳定的 semantic 推理。

## 10. 完成判定

可以认为本轮修改完成，当且仅当满足：

1. 三组实验都能稳定输出 JSON，`parse_ok_rate >= 0.95`。
2. semantic 不再产生大量伪 Android 组件作为 evidence。
3. RAG 每条样本至少检索到 2 条行为规则。
4. RAG 的 `recall_benign` 不再接近 0。
5. `pred_S_ratio` 与测试集真实 S 比例 0.5 不再严重偏离。
6. `results.xlsx`、`figures/`、`rules.csv` 或等价规则库导出、`prompt.md` 都能服务最终报告。

## 11. 最推荐的默认路线

如果不再额外纠结，建议按这个路线执行：

1. P0 全部改完，跑 30 条确认 semantic JSON 恢复。
2. RAG 改 bucketed retrieval，强制 `feature_top_k=2`、`rule_top_k=3`。
3. 知识库补充 B 规则、Context-dependent 规则和训练池统计。
4. semantic 改为中性结构，保留旧 semantic 作为失败案例。
5. 跑 `k=5, test=200` 的最终主实验。
6. 时间允许再补 `k=1/3/5` 与 Core-set 对比。

这条路线最贴合两份审查报告的根因，也最容易在最终报告里讲清楚：不是靠提示词“哄”模型，而是把数据事实、规则检索和上下文约束修正到合理状态。
